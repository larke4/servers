# Servers — Backend епозиторий для работы с android приложением
Таубин Георгий Георгиевич ИКС-431

C++/ZMQ backend-приложение, которое принимает данные со смартфона Android (геолокация, параметры сотовых сетей, трафик), сохраняет их в PostgreSQL, отображает в GUI (ImGui + ImPlot), подгружает OpenStreetMap-тайлы и строит тепловую карту качества сигнала.

## Обзор проекта

Что делает backend:

- Поднимает **ZMQ PULL-сокет** и принимает JSON-пакеты от Android-клиента.
- Парсит:
  - координаты (`latitude`, `longitude`, `altitude`, `accuracy`, `timestamp`),
  - телеком-параметры (LTE/NR/GSM поля: `rsrp`, `rsrq`, `rssi`, `pci`, `earfcn`, `nrarfcn`, `arfcn`, `tac` и т.д.),
  - статистику трафика (`networkUsage`, `topApps`).
- Обновляет состояние в памяти (`RuntimeState`) для GUI.
- Пишет сырые принятые пакеты в `build/received_android.json`.
- Опционально пишет нормализованные записи в PostgreSQL.
- В GUI показывает:
  - текущие данные,
  - графики сигналов по сотам,
  - карту OSM и тепловой слой.

## Архитектура

Проект разделён на логические слои (практика «рефакторинг» соблюдена):

- **yadro** — структуры данных, runtime state, JSON-парсер, генерация теплокарты.
- **zmq/curl** — ZMQ-приёмник, OSM tile fetch/decode.
- **db** — минимальная обвязка PostgreSQL через libpq.
- **ui** — окна и виджеты интерфейса.
- **gui_app/main** — lifecycle приложения и главный цикл.

Потоки:

1. **Поток сервера**: `RunZmqReceiver(...)` слушает входящий порт, парсит пакеты, обновляет state.
2. **Главный поток GUI**: рендерит вкладки/окна, читает shared state.

## Сборка и запуск

## Требования

- CMake >= 3.16
- C++20 компилятор
- cppzmq / libzmq
- nlohmann_json
- PostgreSQL client (libpq)
- SDL2 + GLEW + OpenGL
- libcurl (для OSM-тайлов)
- Threads

## Быстрый старт (пример)

```bash
cmake -S . -B build
cmake --build build -j
./build/rework_app
```

По умолчанию приложение:

- стартует GUI-окно **1400x900**,
- поднимает приёмник данных на порту **5558**,
- использует второй порт **5559** для команд фильтрации.

## Формат входных данных

Поддерживаются как минимум такие варианты ключей:

- `location` или `loc` для координат,
- `telephony` или `mobile_network_data_list.MobileNetworks` для списка сот,
- `networkUsage.topApps` для сетевой статистики.

Сервер гибко обрабатывает разные варианты названий полей (например `latitude` и `Latitude`).

### 6. Местоположение смартфона. Location

Реализовано на стороне backend:

- Приём `Latitude`, `Longitude`, `Altitude`, `Current Time` (+ `Accuracy`) из JSON.
- Накопление траектории (`trajectory`) и текущей точки (`currentUser.location`).
- Логирование и запись входящих данных в файл `build/received_android.json`.

Что предполагается на стороне Android (по заданию) и поддерживается backend-ом:

- `getLastLocation()` / обновления локации.
- Периодическая отправка координат в backend.
- (Доп.) работа через фоновый сервис — серверный протокол к этому готов.

---

### 7. Работа с сокетами (ZMQ). Передача данных от Android к PC

Реализовано:

- Серверная часть на ZMQ (`PULL`) принимает данные от Android.
- Отдельный канал команд (`PUSH`) для отправки управляющих фильтров (`gps/lte/nr`).
- Подсчёт количества пакетов и объёма принятых данных.
- Сохранение каждого полученного JSON-блока в файл.

---

### 8. Backend-сервер на CXX

Реализовано:

- Полноценный backend на C++.
- Потоковая модель: отдельный поток приёмника + GUI поток.
- Общая структура состояния между потоками (`RuntimeState`) для обмена данными.
- Приём телеметрии местоположения и сигналов, визуализация в GUI.

Примечание: авто-reconnect в Android-клиенте — зона ответственности клиента, но серверная часть рассчитана на продолжительный приём потока сообщений.

---

### 9. Android Cellinfo.

Поддержаны поля по LTE/NR/GSM в приходящем JSON, включая:

- идентификаторы соты и сети (`cellId/nci`, `mcc`, `mnc`, `pci`, `tac`),
- радиометрики (`rsrp`, `rsrq`, `rssi`, `sinr`, `cqi`, `timingAdvance`),
- частотные параметры (`earfcn`, `nrarfcn`, `arfcn`),
- геоданные (`Latitude`, `Longitude`, `Altitude`, `Time`, `Accuracy`).

---

### 10. Android background service.

Серверная часть совместима с моделью фонового Android-сервиса:

- непрерывно принимает поток телеметрии,
- отображает графики изменений сигналов во времени,
- не требует видимости Android Activity для приёма данных.

---

### 11. Драйв-тест.

Поддержка драйв-теста обеспечена backend-механикой:

- долговременный приём и накопление точек,
- сохранение сырых пакетов в JSON-lines файл,
- визуальный контроль данных во время движения.

---

### 12. Работа с базой данных (PostgreSQL) из С/С++

Реализовано:

- подключение к PostgreSQL через `libpq`,
- создание/проверка таблицы (`ensureTable`),
- вставка каждой новой записи при приёме телеметрии,
- GUI продолжает показывать «текущее состояние» параллельно с записью в БД.

---

### 13. Отрисовка OpenStreetMap карты

Реализовано:

- пересчёт координат и работа с OSM-тайлами,
- загрузка PNG-тайлов через `curl`,
- декодирование в текстуру,
- отображение карты в GUI,
- кэширование тайлов на диске в формате `build/<zoom>/<x>/<y>.png`,
- использование кэша при повторных запросах,
- асинхронная загрузка (worker pool).

---

### 14. Генерация тепловой карты (Heatmap)

Реализовано:

- генерация теплового слоя по измеренным точкам,
- метод интерполяции IDW,
- ограничение радиуса влияния,
- хранение и обновление массива точек,
- наложение на карту для визуальной оценки качества покрытия.
